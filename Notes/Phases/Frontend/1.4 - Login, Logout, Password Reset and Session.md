## Task 1: Auth Context & Session Management

Create an `AuthContext.tsx` in `src/context/` that manages the entire authentication session across the app. This is separate from the RegistrationContext — RegistrationContext handles the multi-step registration flow, AuthContext handles the logged-in session.

State to track:
```typescript
interface AuthUser {
  clientId: string;          // e.g. "RR-676687-554"
  firstName: string;
  lastName: string;
  email: string;
  phoneLastFour: string;
}

interface AuthState {
  isAuthenticated: boolean;
  user: AuthUser | null;
  token: string | null;
  isLoading: boolean;        // True while checking stored session on app load
  lastActivity: number;      // Timestamp of last user interaction
}
```

Context functions:
- `login(email, password): Promise<LoginResult>` — calls mock API, stores token and user, sets isAuthenticated to true
- `logout()` — clears all auth state, removes stored token, navigates to /login
- `refreshSession()` — updates lastActivity timestamp (called on user interactions)
- `isSessionExpired(): boolean` — checks if 30 minutes have passed since lastActivity

Token storage:
- Store the JWT token in memory (a ref or state variable) as the primary storage — NOT in localStorage (security best practice)
- Optionally store a "session active" flag in sessionStorage so the session survives page refreshes within the same browser tab but NOT across tabs or after the browser is closed
- On app load: check sessionStorage for the flag, if present attempt to restore the session (mock a token refresh call), if absent redirect to login

Wrap the entire app (including auth pages) with AuthProvider so login state is available everywhere.

## Task 2: Inactivity Timer & Auto-Logout

Build an inactivity tracking system inside the AuthContext (or as a separate hook `useInactivityTimer`).

How it works:
- Track user activity by listening to: mousemove, mousedown, keydown, touchstart, scroll events on the document
- Throttle the activity tracker (only update lastActivity once every 30 seconds to avoid excessive state updates)
- Check every 60 seconds whether 30 minutes have passed since lastActivity

Warning modal at 25 minutes:
- At 25 minutes of inactivity, show a modal warning: "Your session will expire in 5 minutes due to inactivity."
- Modal contents:
  - Heading: "Session Expiring Soon"
  - Message: "You have been inactive for 25 minutes. Your session will end in {countdown} to protect your account."
  - Live countdown timer showing minutes and seconds remaining (e.g. "4:32")
  - Two buttons: "Stay Logged In" (primary) and "Log Out Now" (secondary/ghost)
  - "Stay Logged In" resets the inactivity timer and closes the modal
  - "Log Out Now" calls logout immediately

Auto-logout at 30 minutes:
- If the warning modal is ignored and 30 minutes is reached, auto-logout
- Clear the session, navigate to /login
- Show a toast notification on the login page: "You have been logged out due to inactivity."

The inactivity timer should:
- Only run when the user is authenticated (do not track activity on auth pages)
- Pause when the browser tab is hidden (Page Visibility API) — resume timer check when tab becomes visible again
- Clean up all event listeners on unmount

Create a `SessionWarningModal.tsx` in `src/components/ui/`:
- Uses the existing design system (Card, Button components)
- Overlay backdrop (semi-transparent dark)
- Centred modal with accessible focus trapping
- aria-role="alertdialog"
- aria-describedby linking to the warning message
- Escape key triggers "Stay Logged In" (safe default)

## Build notes — what actually happened

Built and verified 2026-05-27 against the live `frontend/app/` ("Modern Jurist" skeuomorphic) build. The spec is a generic template; these are the adaptations the real build required (see also [[1.3 - Registration Flow]] for the reused pieces).

**Adaptations from the literal spec**
- Pages live in `src/routes/`, not `src/pages/`.
- Icons are **Material Symbols** via the `<Icon name>` component — no `lucide-react` was added. `Mail` → `<Icon name="mail">`, checkmark → `<Icon name="check" fill>`.
- Modals use the existing **`@floating-ui/react`** pattern (FloatingPortal + FloatingOverlay `lockScroll` + FloatingFocusManager `modal returnFocus` + `useRole role="alertdialog"`), mirroring `MobileDrawer.tsx`. No hand-rolled focus trap. Modal panel is a `div.skeuo-card` (not `Card`, which isn't a `forwardRef`).
- There was no hardcoded "Hi, Sarah" header. The greeting is built in `Dashboard.tsx`; it now reads `useAuth().user?.firstName`.
- The build only had a Log Out control in `MobileDrawer` (a no-op). Per Brad: **added a Log Out row to the desktop `SideNav` footer** and wired both to `requestLogout()`.
- Routes: the build uses `/chat` (kept) and `/faq`, not `/messages`. Protected set = dashboard, claims, claims/:id, documents, chat, profile, faq.
- Themes are **Light / Dark / System** only (no separate High-Contrast mode); verified Light + Dark/System and the 3 text sizes. All new UI uses semantic tokens so it tracks the theme automatically.
- Test user aligned to the existing mock CRM record **Sarah Holden** (`RR-676687-554`, phone `7890`) rather than the spec's "Mitchell", so greeting + dashboard data stay consistent.
- `/` keeps the existing **Splash** screen, which now routes to `/dashboard` or `/login` based on session (guarded on `isLoading`).
